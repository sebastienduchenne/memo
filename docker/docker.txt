*** Défintion
- plate-forme logicielle qui permet de concevoir, tester et déployer des applications rapidement
- fait de la conteneurisation = peut empaqueter une application et ses dépendances nécessaires à son fonctionnement dans un environnement isolé appelée conteneur (une image docker)
- éléments : bibliothèques, outils système, code et environnement d'exécution
- Un conteneur n'inclut pas d’OS, mais s'appuie sur les fonctionnalités de l’OS fournies par la machine hôte Linux :
    -Cgroups : groupes de controles, limitent l’accès aux ressources mémoire, CPU et E/S, ce qui empêche les besoins en ressources d’un processus d’affecter d’autres processus en cours d’exécution
    -namespace : limitent un processus et ses processus fils à une section spécifique du système sous-jacent
- les VM virtualisent le matériel d'un serveur et les conteneurs virtualisent l'OS d'un serveur


*** Avantages
- accroît la flexibilité et la portabilité d'exécution d'une application, laquelle va pouvoir tourner de façon fiable et prédictible sur une grande variété de machines hôtes
- les applications ayant des exigences différentes peuvent être exécutées indépendemment les unes des autres sans que l’overhead d’un système invité séparé doive être accepté
- économe en ressources
- délais entre écriture du code et MEP court


- Apps always run in the same environment (local computer, live server).
- Apps are sandboxed to keep them separate and avoid potential conflicts.
- Apps are easy to share with all their dependencies.

-les apps ont leur propre hote
-lancer test d'intégration indépendamment

- Les conteneurs logiciels permettent aussi d’utiliser des applications sur plusieurs plateformes et dans différentes infrastructures sans avoir à les adapter spécifiquement aux configurations matérielles ou logicielles des différents systèmes hôtes.
Docker utilise des images portables de logiciel de conteneurs. Les images de conteneurs contiennent des applications propres, y compris toutes les bibliothèques, les fichiers binaires et les fichiers de configuration nécessaires à l’exécution des processus d’application encapsulés et n’imposent donc que des exigences minimales au système hôte en question. Ceci permet de déplacer un conteneur d’application entre différents systèmes Linux, Windows ou MacOs sans configuration supplémentaire, à condition que la plateforme Docker ait été installée en tant que couche abstraite.


*** Image docker
-template avec des instructions pour créer un conteneur
-chaque image est une superposition de couche. Cela permet de restaurer une version précédente
-on utilise un dockerfile pour créer une image


*** Docker engine
= application client-serveur. 3 composants :
    -server = le daemon docker : processus en arrière plan qui gère des objets Docker (images, conteneurs, volumes et réseau). Ecoute l'API
    -API-REST : interface pour communiquer avec le daemon
    -docker client CLI : programme client du point de vue de Docker qui intéragit avec le daemon via l’API-REST
-> server C API C CLI


*** Docker hub
= registre basé sur le Cloud pour les référentiels (dépôts) de logiciels


*** Dockerfile
= ensemble d’instructions et de métadonnées qui permet de construire une nouvelle image à partir d’une autre
.peut être optimisé pour gagner du temps

*** conteneur
= instance d'une image et configuration du conteneur (mappage de port, dossiers à partager, ...)
- a un cycle de vie
- léger car pas besoin d'un hyperviseur
- connectable à un ou plusieurs réseau
- par défaut, les conteneurs sont isolés les uns des autres et de la machine. On peut configurer des réseaux
- par défaut, le conteneur peut se connecter au réseau de l'hote

* docker run -i -t ubuntu /bin/bash
1- si ubuntu absent : download ubuntu (docker pull ubuntu)
2- création du conteneur (docker container create)
3- allocation filesystem pour le conteneur
4- création d'une interface pour connecter le conteneur à un réseau (avec IP)
5- démarrage du conteneur et exécution /bin/bash
6- si exit, le conteneur stoppe mais n'est pas supprimé


*** services
- dans une application distribuée, les différentes parties sont des services (BD, server, front)
- un service lance une image


*** mode swarm
= cluster de différentes machines (appelé worker, ou node) qui vont fonctionner dans un schéma "master / worker"
- Chaque service est susceptible d'avoir des besoins en ressources différents et peuvent croître à des rythmes différents. En séparant les services dans des conteneurs différents, nous pouvons composer chaque service en utilisant le type d'instance le plus approprié en fonction de différents besoins en ressources. 
- un cluster a un ou plusieurs manager appelés orchestrateurs
- dans un cluster les conteneurs sont appelés services
- utilisation d'un load balancing pour répartir la charge sur les différents noeuds
- dans un cluster : le manager gère différents services/node, qui contiennent chacun un daemon gérant plusieurs conteneurs
- un conteneur s'exécutant dans un service s'appelle une tache

       manager
       /    \
daemon      daemon
conteneur   conteneur
conteneur   conteneur


-si plusieurs manager, un seul (le leader) gère la configuration du cluster et les autres (followers, status reachable) sont disponibles pour remplacer le leader en cas de défaillance
-docker-compose : définir et lancer plusieurs composants facilement

* stratégies
-noeud le plus vide : remplit les machines les moins utilisés
-global : chaque machine a une seule instance du conteneur spécifié




.Single Point Of Failure = si le master tombe il sera automatiquement remplacé par un des noeuds qui deviendra le nouveau master
.docker compose = lancer plusieurs conteneur à la suite, grâce à un fichier de déploiement au format YAML précisant tous les conteneurs et leurs options exécutable en une seule fois.
.google cAdvisor : Fournit aux utilisateurs de conteneurs une bonne compréhension sur l'utilisation des ressources et des performances générales sur les conteneurs en cours d'utilisation. Il s'agit d'un processus daemon qui collecte, aggrege, opère et exporte les informations des conteneurs en cours.

-> on déploie plusieurs conteneurs (BD + log + app + serveur)

Vous avez surement plusieurs projets qui utilisent MySQL. Il est possible d’installer MySQL sur chacun de vos conteneurs, ou bien d’avoir un conteneur dédié. De cette manière, tous vos projets stockeront leurs données sur le même conteneur. Cela permet de rassembler un maximum de données, et de faciliter la gestion (un seul et même conteneur à lancer quel que soit le projet, à condition que la version de la BDD voulue soit similaire).
